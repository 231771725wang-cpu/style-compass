# 风格罗盘（style-compass）红队测试清单

这份清单服务于开源前验收与后续复跑，不是为了证明“脚本能跑”，而是为了回答一个更直接的问题：

这个 skill 会不会让真实用户迷路、失望，或者觉得“看起来很多，实际上没用”。

## 目标

- 同时打三类风险：`不会用`、`推荐不准`、`产物没价值`
- 用真实用户口吻做压力测试，不用内部工程术语包装 prompt
- 保留证据，支持修复后回归复跑

## 固定评分维度

每条 eval 都按这 5 个维度写人工评审：

1. `发现性`
2. `推荐可信度`
3. `候选区分度`
4. `产物可交接性`
5. `诚实度 / 不装懂`

建议每项都写一句“好在哪里 / 坏在哪里”的证据，不要只打感受分。

## 风险级别定义

- `P0`
  会直接让用户失去信任，或非常容易被公开避雷。
  例：推荐明显跑偏、完全没引导到总览、产物几乎不可用、缺信息时还强装懂。

- `P1`
  不会立刻翻车，但会显著降低推荐意愿。
  例：三张卡没区别、解释空泛、handoff 不够落地、用户仍然不知道下一步。

- `P2`
  基本可用，但体验明显粗糙。
  例：文案重复、步骤略绕、新手负担偏高。

- `P3`
  细节优化项。
  例：措辞、排序、轻微冗余。

## Stop-Ship 规则

- 任意 `P0`：开源前必须先修。
- 同一类问题出现 2 次以上 `P1`：默认列入首轮修复清单。
- `with_skill` 相比 `baseline` 没有明显更好：该 eval 记为失败或存疑。

## 每条 eval 必须留下的证据

- 原始用户说法：`prompt.txt`
- 结构化元数据：`eval_metadata.json`
- `with_skill/outputs/`
- `baseline/outputs/`
- 人工评审：`review.md`
- 结构化结论：`verdict.json`

## 人工评审时重点看什么

### A. 首次使用与发现性

- 有没有自然把用户带到总览页
- 用户是否 10 秒内知道这个 skill 的价值
- 是否过早堆命令、参数和脚本细节

### B. 推荐质量与风格区分度

- 3 张风格卡是否真的不同
- 用户点名风格是否被保留
- 推荐理由是不是“同义改写 + 空形容词”

### C. Refactor

- 是否会在缺截图、缺 repo 信号时诚实降置信度
- 是否会越界，把视觉升级说成架构改造
- `UI-REFACTOR.md` 是否真的能交接

### D. 产物价值

- `DESIGN.md` / `UI-REFACTOR.md` / handoff prompt 是否可直接给下游继续做
- 是否空泛、重复、模板味太重
- 是否需要人类再翻译一遍才能继续执行

## 复跑建议

- 每次大改后，至少重跑全部 `P0/P1` 和每组 1 条代表用例
- 若新增入口、推荐逻辑或 prompt 逻辑，优先补对应组别的 eval
- 如果某类失败模式反复出现，升级成下一轮固定回归集
