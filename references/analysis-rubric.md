# 项目画像分析规则

先把项目归到“什么产品”与“什么气质”，再谈风格。

## 一、仓库结构信号

### 1. 页面类型

优先识别：

- `marketing`：Hero、大图、案例、CTA、定价
- `dashboard`：侧边栏、表格、状态卡、筛选区
- `editor`：画布、属性面板、工具栏、图层区
- `docs`：文档导航、章节目录、示例代码
- `commerce`：商品图、购买流程、价格、规格
- `media`：封面图、内容流、栏目化排版

### 2. 信息密度

- `low`：大留白、少控件、重点在品牌或叙事
- `medium`：有功能块，但阅读与操作平衡
- `high`：面板密集、数据多、连续操作强

### 3. 产品气质

- `tooling`：效率、准确、专业、控制感
- `editorial`：阅读感、层次、排版节奏
- `brand`：态度、形象、视觉记忆点
- `commerce`：转化、商品聚焦、行动驱动

## 二、截图信号

### 1. 亮暗倾向

- `light`：白底、浅灰、留白、轻感
- `dark`：黑底、深灰、霓虹、工具感
- `mixed`：浅深并置，靠大图或章节切换气氛

### 2. 图片权重

- `low`：几乎不依赖大图，重组件与排版
- `medium`：局部产品图或插图增强表达
- `high`：摄影、海报、封面图主导版面

### 3. 排版张力

- `precise`：克制、精准、几何、秩序
- `editorial`：杂志感、字重层次强
- `aggressive`：超大标题、广告化、态度明显
- `friendly`：圆润、轻松、亲和

## 三、已有项目改造时额外补的信号

### 1. 当前 UI 问题

优先沉淀：

- 层级是否混乱
- 密度是否失衡
- 组件语言是否不统一
- 品牌感是否不足
- 移动端是否压缩失败

把这些问题写进 `current_ui_findings`，不要只写成“感觉不够高级”。

### 2. 现有页面清单

把仓库里最重要的页面职责写进 `existing_pages`，例如：

- `dashboard`
- `editor`
- `docs`
- `marketing`
- `settings`

### 3. 组件清单

把当前最常出现的结构组件写进 `component_inventory`，例如：

- `sidebar`
- `topbar`
- `data table`
- `card`
- `form`
- `modal`

### 4. 改造边界

至少选清楚下面三档中的一档，并写入 `refactor_scope`：

- 仅视觉层
- 允许布局调整
- 允许组件重组但不改核心信息架构

## 四、风格匹配原则

### 工具/控制台/编辑器

优先考虑：

- Linear
- Supabase
- Sentry
- Vercel
- Cursor
- HashiCorp

避免直接推：

- Nike
- Airbnb
- Pinterest

除非用户明确要品牌化营销感。

### 品牌页/营销页

优先考虑：

- Apple
- Nike
- Stripe
- Framer
- Tesla
- Shopify

若产品仍然很工具型，至少保留一个稳妥工具风作对照项。

### 文档/开发者官网

优先考虑：

- Vercel
- Mintlify
- Replicate
- MongoDB
- Cal.com

### 画布/创意工具

优先考虑：

- Figma
- Miro
- Framer
- Apple
- Linear

这里要注意区分：

- 如果是“专业创作工具”，偏 Figma / Linear / Apple
- 如果是“协作白板”，偏 Miro
- 如果是“展示型创意工具”，偏 Framer

## 五、多样性约束

每次推荐 3 个风格时，不要全是同一审美簇。

至少满足以下之一：

- 分类不同
- 一个稳妥、一个相邻、一个对照
- 一个偏工具效率、一个偏品牌表达、一个偏折中路线

## 六、风险提示模板

### 典型风险

- “品牌感很强，但会压缩工具效率”
- “深色工具风很稳，但营销张力不足”
- “摄影权重高，若素材弱会显得空”
- “编辑感强，但实现细节要求更高”
- “过于极简，复杂功能页面会显得薄”
